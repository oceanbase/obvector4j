Three sibling packages under `com.oceanbase.obvector4j` with a one-way dependency direction: `schema` → `util`, `version` is standalone.

- `schema/`: immutable-ish value objects built via fluent setters (`Name(...)`, `DataType(...)`, `IsPrimary(...)`, etc.) that all extend the abstract `Visitable` base class. Each overrides `visit()` to emit an SQL fragment; `ObCollectionSchema.visit()` aggregates field definitions, primary-key clauses, and optional `IndexParams` into a single table-definition string. `ObFieldSchema` validates constraints (e.g. `VARCHAR` needs `maxLength`, `FLOAT_VECTOR` needs `dim`; auto-increment cannot be nullable) before rendering. `IndexParam` renders HNSW index options (`WITH(m=..., ef_construction=..., ef_search=..., lib=vsag, distance=..., type=hnsw)`); `IndexParams` joins multiple such definitions. `DataType` enum maps SDK types to MySQL-compatible column names via `Convert2MySQL()`.
- `util/`: stateless helper classes (`JdbcTypeMapper`, `VectorMetric`) with private constructors used only for static methods — `JdbcTypeMapper.fromJdbc` converts JDBC `Types` + typeName back to `DataType`, and `VectorMetric` resolves metric aliases (`ip`/`cosine`/`l2`) to OceanBase distance functions and formats float arrays as literals.
- `version/`: `OceanBaseVersion` parses both `OceanBase-vX.Y.Z` and plain `X.Y.Z` strings, exposes `isAtLeast` / `compareTo`, and ships pre-defined constants like `HYBRID_SEARCH_SQL_MIN` for feature gating.

No runtime framework or reflection is used; the module is pure POJOs + regex, intended to be consumed by higher-level client/connection layers to build CREATE TABLE statements.
