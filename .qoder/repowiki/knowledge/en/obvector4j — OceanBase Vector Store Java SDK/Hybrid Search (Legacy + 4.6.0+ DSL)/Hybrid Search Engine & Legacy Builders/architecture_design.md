Two-layer design inside `com.oceanbase.obvector4j.hybrid`:
- Builder layer: `AbstractHybridSearchBuilder<T>` holds shared `ObVecClient`, table/output-field state and the `resolveOutputFields()` routine (field trimming + type inference via `client.inferColumnDataType`). Concrete builders `HybridTextVectorSearchBuilder` (fulltext+vector) and `HybridScalarVectorSearchBuilder` (scalar+vector) validate parameters and call back into `ObVecClient`.
- Execution layer: `HybridSearchEngine` is the single runtime entry point. It receives a `VersionSupport` probe and branches at runtime — if OceanBase ≥ 4.6.0 supports `HYBRID_SEARCH(TABLE t, ?)` with a JSON DSL, it delegates to `HybridSearchDslBuilder`/`HybridSearch460Support`; otherwise it falls back to legacy SQL: separate `MATCH(...) AGAINST(...)` fulltext query and distance-based vector query, then combines them in-memory via `HybridResultMerger` using Reciprocal Rank Fusion (`1/(k+rank)`).
- Supporting utilities: `OutputFieldValidator` centralizes null/empty field sanitization and returns a `ValidatedFields` value object; `HybridResultMerger` is package-private and implements the RRF merge/sort/top-k cut.
Dependency direction is one-way: builders → engine → JDBC; engine depends on `v460` DSL only through the injected `VersionSupport` interface, keeping the legacy path compilable against older runtimes.
